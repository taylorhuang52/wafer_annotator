Wafer Defect Demo — 架構筆記

本地端不用裝 PyTorch，
是因為它被搬到了另一個房間

這份筆記說明：為什麼把 AI 模型部署成 Hugging Face Space 之後， 你本機的 venv 可以從一大包瘦成幾十 MB， 以及這中間資料到底是怎麼流動的。

把它想成一片 wafer map：大部分的 die 良好（■ pass）， 少數幾顆才需要送進更複雜的分析流程（■ 需要 AI 模型）。 你的本地端程式，做的正是同一件事。

問題出在哪

torch 佔掉的，不是你程式的重量，是「以防萬一」的重量

你原本的本地 app.py 為了能做推論，得把整套 torch / torchvision 裝進 venv—— 即使 90% 以上的 wafer 良率一看就過關、根本用不到模型。 這包依賴通常就佔掉 venv 一大半的體積。

含 torch 的本地 venvflask + numpy + scipy + torch + torchvision + huggingface_hub
拆分後的本地 venvflask + numpy + scipy + pillow + requests

（比例僅示意：torch/torchvision 的 wheel 本身動輒數百 MB 起跳，其餘套件加總通常不到它的十分之一。）

資料怎麼流動

三個角色，各管各的事

關鍵想法：把「跑模型」跟「處理業務邏輯／畫面」拆成兩個完全獨立的服務， 中間只靠一個很小的 JSON 訊息溝通。

整趟旅程裡，「你的電腦」從頭到尾沒有執行任何 Python——運算全部發生在雲端的兩個容器裡。

回到你的問題

Hugging Face Space 裡跑的是 venv 嗎？

更精確地說是 Docker 容器，觀念上比 venv 更徹底一層：

venv

同一台機器，隔開套件

  • 還是同一個作業系統、同一個檔案系統
  • 只是把 Python 套件安裝路徑獨立出來
  • 兩個 venv 之間仍共用 CPU / 記憶體 / 硬碟
docker container

連「房間」都是分開的

  • 每個 Space 都是自己專屬的最小系統環境
  • 檔案系統、Python 環境、依賴套件完全獨立
  • 兩個容器唯一的溝通方式就是網路上的 HTTP 請求

所以你的模型 Space 容器裡確實裝了 torch、也確實在裡面跑推論， 但那個「肥大」被永久關在那個容器裡，跟你本地端、跟前端 Space，都毫無關係。

為什麼這樣拆是對的

只有真正需要重型工具的那一步，才配得到重型工具

這其實跟你 app.py 裡本來就有的良率規則邏輯是同一種精神： 良率 ≥ 95% 的 wafer 用規則就能判定「無缺陷」，完全不必浪費一次模型推論。 拆分成兩個 Space，只是把這個「按需求動用重型資源」的原則， 從「要不要跑模型」延伸到「要不要在這台機器上裝模型的依賴」。

代價在哪：多了一次網路來回，加上免費 Space 閒置會 sleep， 喚醒模型 Space 第一次呼叫可能要等 20–30 秒。用體積換來的，是速度上的一點點延遲—— 對一個不是即時系統的內部分析工具來說，這筆交易通常划算。
