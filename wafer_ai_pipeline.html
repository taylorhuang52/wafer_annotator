01

模型本體：一顆只認得 8 種缺陷的 ResNet34

辨識引擎是誰、吃什麼格式、吐出什麼結果

這套系統的辨識核心，是一顆基於 ResNet34 架構的卷積神經網路，由 Hugging Face 上 radai-agent/radai-wm811k-defect-detection 這個公開模型提供，用 WM-811K（業界最大的公開晶圓圖資料集，17.2 萬筆）訓練而成。

模型的輸入固定是一張 64×64 的灰階陣列，數值只有三種：0 代表背景（wafer 邊界外、沒有 die 的地方）、1 代表良品 die、2 代表不良 die。不管你丟進去的原始資料是什麼尺寸的 wafer，系統都會先縮放、正規化成這個固定格式。

輸出則是一個 8 類的機率分佈（softmax），加總必為 100%：

Pattern典型特徵
Center缺陷集中在中心
Donut環狀分佈，中心與邊緣正常
Edge-Loc邊緣局部異常
Edge-Ring沿邊緣整圈異常
Loc非邊緣的局部群聚
Random隨機散佈
Scratch線狀刮痕
Near-full近乎整片異常
為什麼是 8 類，沒有「正常 / None」？
原始 WM-811K 其實有 9 類，但 None（無特定圖案）佔了整體 85%。如果照原比例一起訓練，模型會傾向無腦猜 None 就能拿到 85% 準確率，反而學不好真正需要辨識的缺陷 pattern——這是分類任務中經典的 class imbalance（類別失衡）問題。這顆模型選擇拿掉 None，讓 8 類缺陷彼此數量接近、訓練更聚焦。也因此，「這片是不是正常的」這件事，不該問模型，而是要用良率數字直接判斷——這正是下面第 03 節「良率規則判定」存在的原因。
02

兩種輸入路徑：圖片 vs. Test Log

同一顆模型，兩種完全不同的資料前處理方式

系統支援兩種完全獨立的輸入來源，殊途同歸地變成同一種 64×64 陣列格式，再送進模型。

路徑 A · 單張圖片
  • 使用者上傳 PNG / JPG
  • 轉灰階，縮放成 64×64
  • 依最大值正規化到 0~1
  • 直接送進模型
路徑 B · Test Log CSV
  • 解析 XAdr / YAdr / Bin#
  • 還原成真實的 die 座標網格
  • Bin#==1 → 良品，其他 → 不良
  • 縮放成 64×64 才送進模型

路徑 B 是這套系統真正的核心價值：它不需要人工先把 wafer 畫成圖，而是直接從機台吐出的原始 test log 讀取 Serial#, Site#, Bin#, SBin#, XAdr, YAdr 這幾欄，逐顆 die 還原出一個 bin_map 陣列——這比任何截圖或拍照都精確，因為它是機台實際量測的座標數字，不會因為畫圖、壓縮、縮放而失真。

解析過程中還會順便做一件事：從 CSV 表頭抓出 IR0(8)、VF1(17) 這類「測項名稱(Bin 編號)」的對照表，讓系統知道每一顆 fail die 具體是哪一個測項造成的，而不只是「有沒有 fail」。

Serial#,Site#,Bin#,SBin#,XAdr,YAdr,...
0,7,9,9,20,6,...   → Bin 9 = DIR0 測項 fail
1,4,1,1,21,6,...   → Bin 1 = 良品
03

良率規則判定：不是每片都要跑模型

用一個簡單、確定的數字，取代模型不擅長判斷的事

因為模型從未學過「正常」這個類別，如果把良率很高、幾乎沒有缺陷的 wafer 硬塞給它，它只能在 8 種「都是缺陷」的選項裡勉強選一個，通常會伴隨很低的信心分數。與其讓模型硬猜，不如用一條更可靠的規則先過濾：

Step 1
計算良率
Pass 顆數 ÷ 總顆數
→
Step 2 · 規則
良率 ≥ 門檻？
門檻可調，預設 95%
→
是
直接標記 None
不呼叫模型
→
↓ 否
Step 3
送進 ResNet34
輸出 8 類機率排序
這個門檻是人工工程判斷，不是模型算出來的。 良率≥門檻的 wafer 完全不會被模型看到、也不會產生任何機率數字——避免介面上出現「假造的信心分數」誤導判讀。門檻值可以在介面上即時調整，每一批批次分析都會重新套用當下設定的數字。
04

跨 Wafer 同座標熱點：找出「固定機構」造成的問題

不是每片單獨看，而是把整批疊在一起比對

單看一片 wafer 的缺陷分佈，很難分辨這是隨機顆粒污染還是固定機構造成的系統性問題（例如探針卡某根 pin 位置固定不良、chuck 上有固定污染點）。判斷的關鍵在於：如果同一個 (X, Y) 座標在整批很多片 wafer 上都重複失效，機率上不太可能是巧合。

對齊方式：每片 wafer 的座標系統不一定完全從 (0,0) 開始，系統會先取整批所有檔案裡 XAdr / YAdr 的絕對最小值，建立一個共用的座標網格，再把每片的資料疊上去逐格累加：

for 每一片 wafer:
    for 每一顆 die at (x, y):
        tested_count[y][x] += 1        // 曾經被測到
        if Bin# != 1:
            fail_count[y][x] += 1       // 在這片上 fail

最後 fail_count 除以總片數，就是每個座標的「重複失效比例」。比例越高，顏色越紅，代表越可能是機構性問題，值得排查探針卡、chuck、cassette slot 或 chamber 內部的固定死角，而不是急著找隨機微粒。這個分析同樣適用於單一片 wafer（n=1），此時就單純是那一片自己的 fail die 分佈圖。

05

人機協作：模型判斷錯了，怎麼記下來

Human-in-the-loop：修正不會消失，會被存下來變成未來的訓練材料

模型的判斷不會永遠正確，尤其遇到訓練資料裡少見的圖案組合時，信心分數往往偏低。當工程師發現模型判斷錯誤，可以直接在介面上選擇正確的分類並存檔，系統會把原始資料（不是圖片）連同修正結果一起保留下來：

存的內容
  • labels.csv — wafer ID、模型原判斷、人工修正後標籤、主要 fail 測項、時間戳記
  • binmaps/*.npy — 該片 wafer 完整的原始 Bin# 陣列
為什麼不存圖片
  • 圖片是失真的視覺化，數字才是機台量測的原始事實
  • 存數字陣列可以無損還原成任何格式，也是模型訓練需要的形式
  • 檔案體積小非常多

下拉選單刻意把 「None（無特定圖案）」和「不確定/其他」分開：前者是工程師明確判斷「這片就是正常的」，後者是工程師自己也無法判斷，兩者存進紀錄的意義完全不同——把它們混在一起，將來會汙染訓練資料的品質。

06

未來 Retrain：修正資料要怎麼跟原始資料合併

累積夠了之後，不能只靠這批小資料重新訓練

corrected_labels/ 裡累積的資料通常只有幾十到幾百筆，而 WM-811K 原始訓練集有 17.2 萬筆。如果只拿這一小批去 retrain，模型很容易發生 catastrophic forgetting（災難性遺忘）——在努力學會這幾百筆的同時，忘記了原本從 17 萬筆學到的判斷能力。

格式轉換
Bin 編號 → 0/1/2
修正資料存的是實際 Bin#，訓練前需簡化成模型認得的三值格式
→
資料合併
WM-811K + 修正資料
兩邊都是同一種陣列格式，混成同一份訓練清單
→
取樣加權
修正資料給更高權重
量少，需要被多抽到幾次，才不會被 17 萬筆稀釋掉
驗證是必要步驟，不是選配。 Retrain 後一定要拿一份獨立的驗證集（可從 WM-811K 切出一部分）比較新舊模型的準確率，確認新模型沒有變差，才正式換掉線上使用的權重檔。模型檔案建議帶版本號（例如 _v2_20260701），方便隨時退回舊版本。

這個迴圈裡，機器負責的是規模——一次處理整批 lot、跑一致的判斷標準、不會累；人負責的是判斷力——決定良率門檻該設多少、修正模型看走眼的地方、決定累積夠不夠拿去 retrain。兩邊分工清楚，系統才會隨著時間越用越貼近實際產線的狀況，而不是停在剛下載模型那一刻的準確率。
